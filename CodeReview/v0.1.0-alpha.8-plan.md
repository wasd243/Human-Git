# HumanGit v0.1.0-alpha.8 Plan & Code Review

## 1. Project Status Summary
- **Current State:** The project has successfully integrated the `git2` crate for native Git operations, moving away from fragile shell command parsing. The background daemon uses `notify-debouncer-mini` for efficient filesystem event debouncing, and `r2d2` ensures thread-safe SQLite operations using WAL mode. The UI is highly responsive, featuring a modularized CSS architecture, sliding animations, and dynamic state updates via Tauri's managed state and IPC events.
- **Key Achievements:** Robust `run_shadow_sync` logic that preserves IDE markers via stashes, zero-overhead diff stats calculation, and decoupled UI-to-backend communication. 

## 2. Module Implementation Plans (Rust with `git2`)

### 2.1 `git init` Module Plan
*Completed*

**Target File:** `src/modules/operations/init.rs`
- **Objective:** Allow users to initialize a new Git repository natively.
- **Implementation Strategy:**
  1. Use `git2::Repository::init(path)` to create a new repository in a designated folder.
  2. **Initial Commit:** Create an empty initial commit (or commit a basic `.gitignore` and `README.md`) immediately after initialization to establish the `HEAD` reference. This prevents "unborn branch" errors during subsequent branch operations.
  3. Return a success payload to Tauri so the UI can seamlessly transition into monitoring mode.
- **Error Handling:** Gracefully catch and report if the directory already contains a repository (`.git` exists).

### 2.2 `git stage changes` Module Plan
*Completed*

**Target File:** `src/modules/operations/add.rs`
- **Objective:** Stage specific or all untracked/modified files into the Git index.
- **Implementation Strategy:**
  1. Open the repository: `let repo = git2::Repository::discover(path)?;`
  2. Open the index: `let mut index = repo.index()?;`
  3. **Stage Specific Files:** Iterate over incoming paths and use `index.add_path(Path::new(&relative_path))?;`
  4. **Stage All Changes:** Use `index.add_all(["*"].iter(), git2::IndexAddOption::DEFAULT, None)?;`
  5. **Persist:** Write the index to disk using `index.write()?;`
- **Integration:** Expose a Tauri command `stage_files(paths: Vec<String>)` to let the UI control granular staging.

### 2.3 `git push` Module Plan
*Completed*

**Target File:** `src/modules/operations/push.rs`
- **Objective:** Push local commits to a remote repository securely.
- **Implementation Strategy:**
  1. Open the repository and locate the remote: `let mut remote = repo.find_remote("origin")?;`
  2. Configure `PushOptions` with a `RemoteCallbacks` struct.
  3. **Authentication (Crucial):** `git2` does not automatically use CLI credentials. Implement the `credentials` callback inside `RemoteCallbacks` to handle SSH (via `git2::Cred::ssh_key_from_agent`) or HTTPS (using standard Git Credential Helpers).
  4. Perform the push: `remote.push(&["refs/heads/main:refs/heads/main"], Some(&mut push_options))?;`
- **Concurrency:** Because pushing involves network I/O, execute this entire flow inside a `tokio::task::spawn_blocking` closure to prevent stalling the async Tauri/Tokio runtime.

## 3. Architectural Advice for v0.1.0-alpha.8

1. **Offload Blocking `git2` Operations:**
   - **Context:** The `git2` crate is purely synchronous.
   - **Advice:** Heavy repository traversals, massive diffs, and network operations (fetch/push) should be wrapped in `tokio::task::spawn_blocking`. Currently, operations like `run_shadow_sync` run on the main async threads. Moving them to blocking threads will keep the UI and watcher loop maximally responsive.

2. **Centralize Error Handling:**
   - **Context:** The codebase currently mixes `anyhow::Result`, `std::io::Result`, and string-based errors across modules.
   - **Advice:** Populate the empty `src/modules/shared/errors.rs` file with a unified `thiserror` enum (e.g., `HumangitError`). This enables typed, predictable error bubbling (`?`) and simplifies mapping backend errors to frontend UI notifications.

3. **Global Git Configuration Integration:**
   - **Context:** The author signature is currently hardcoded as `Signature::now("HumanGit", "humangit@system.local")`.
   - **Advice:** Use `git2::Config::open_default()` to dynamically read the user's global `user.name` and `user.email`. Fallback to the HumanGit placeholder only if the local/global config is missing. This ensures commits made by the app are properly attributed to the developer.

4. **Credential Management Strategy:**
   - **Context:** Future push/fetch actions require robust auth.
   - **Advice:** Instead of writing custom password prompts, leverage the OS's native credential manager. Integrate `git2`'s credential helper support so that HumanGit inherits the existing SSH agents or Windows/macOS credential stores already configured by the user.
